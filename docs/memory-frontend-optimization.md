# Memory 前端页面优化设计

> 版本：v1.0  
> 日期：2026-07-14  
> 范围：`static/index.html`、`static/memory.js`、`static/wiki-graph.js`、`static/style.css`、`static/i18n/{zh,en}.json` 及 `/api/memory/*` 的前端消费方式  
> 目标：把当前“以知识图谱为主的展示页”升级为可浏览、可检索、可治理、可解释的记忆工作台。

---

## 1. 结论摘要

当前 Memory 页面已具备以下基础：

- 新建、导入、查看、编辑、软删除记忆文件。
- Canvas 知识图谱及节点点击打开文件。
- LLM 整理预览、勾选、应用流程。
- 自动上下文注入和自动沉淀的聊天侧提示。
- 深浅主题、响应式基础样式和中英文文案体系。

但页面主体实际上只展示一张占据约 `80vh` 的知识图谱，文件列表、搜索、目录树、类型筛选、候选审核、来源与注入状态均未形成可用的信息架构。`loadMemoryFiles()` 只更新文件数量与刷新图谱，没有渲染文件，因此用户无法在 Memory 页完成日常知识管理。

**推荐方向：采用“列表为主、图谱为辅、详情常驻、治理独立”的四区工作台。**

---

## 2. 当前实现与证据

### 2.1 页面结构

Memory 页目前由标题、三个操作按钮、状态文本和单一图谱面板组成：

- 页面入口与操作栏：`static/index.html:718-735`
- 图谱是页面唯一内容区域：`static/index.html:736-744`
- 图谱最小高度为 `80vh`：`static/style.css:5688-5701`

当前缺失：

- 文件列表容器。
- 搜索输入框。
- 目录树与面包屑。
- 类型、来源、注入策略、置信度筛选。
- 排序、分页或虚拟列表。
- 候选记忆和沉淀任务入口。
- 页面级加载、错误、空状态。

此外，`indexMemoryFiles()` 会查找 `#btn-memory-index`，但当前页面没有这个按钮，因此手动索引逻辑属于不可达入口：`static/memory.js:93-108`、`static/index.html:718-744`。

### 2.2 数据加载

`loadMemoryFiles()` 会调用 `/api/memory/files`，但只执行两件事：更新状态文本并刷新图谱，没有保存或渲染返回文件：`static/memory.js:20-31`。

状态栏存在以下问题：

- 文案硬编码为英文 `files`、`last updated`，未走 i18n：`static/memory.js:8-18`。
- 只有总数和最后更新时间，无法体现索引状态、孤立节点、待审核候选、自动注入状态。
- 加载失败只写 `console.error`，页面无反馈：`static/memory.js:28-30`。

### 2.3 文件操作

现有查看、编辑、删除链路完整，但错误反馈不一致：

- 查看请求失败或非 2xx 时直接返回，无用户提示：`static/memory.js:33-50`。
- 编辑器异步加载失败被空 `catch` 吞掉：`static/memory.js:174-180`。
- 保存没有检查 `resp.ok`，即使服务端返回 400/500，仍关闭编辑器并刷新：`static/memory.js:194-208`。
- 删除流程相对完整，包含确认、错误 toast 和刷新：`static/memory.js:64-86`。

### 2.4 整理复核

整理流程是当前页面最成熟的交互：

- 有加载状态和 `aria-live`：`static/memory.js:336-360`。
- 有动作徽标、默认选择、理由、目标文件和内容预览：`static/memory.js:362-403`。
- 支持部分失败展示：`static/memory.js:412-458`。
- 有减少动画设置：`static/style.css:429-434`。

仍有不足：

- 无按风险分类或批量选择规则。
- 合并/改写仅预览前 1200 字，不能查看原文与新文 diff：`static/memory.js:379-381`。
- `listEl.dataset.actions` 保存完整 `new_content`，大批量时会增加 DOM 属性体积：`static/memory.js:397`。
- 应用期间没有防止关闭模态或重复操作的统一 busy 状态。
- 删除、改写、合并均默认勾选；高风险动作不应与安全动作同一默认策略：`static/memory.js:378`。

### 2.5 图谱

图谱具有拖拽、缩放、位置持久化、焦点视觉状态和点击节点打开文件等能力：

- 初始化和 ARIA 基础：`static/wiki-graph.js:71-114`
- 节点位置本地持久化：`static/wiki-graph.js:27-46`
- 点击节点打开文件：`static/wiki-graph.js:740-754`

问题：

1. Canvas 的 `aria-label` 是硬编码英文，未本地化：`static/wiki-graph.js:103-106`。
2. 设置了 `tabindex` 和焦点节点状态，但未发现键盘方向键、Enter 打开、`+/-` 缩放等事件绑定，键盘用户无法操作节点。
3. 力导向模拟为节点两两计算，节点增多后会出现 O(n²) 性能压力；常量为最多 420 次迭代：`static/wiki-graph.js:7-12`。
4. 图谱没有加载、空数据、错误、图例、筛选和节点搜索状态。
5. basename 作为节点 ID 时，重名文件在后端会碰撞，前端无法区分同名路径；该问题需要后端先修复。

## 3. 优化目标

### 3.1 用户目标

用户进入 Memory 页后应能在 10 秒内回答：

1. 当前项目有多少条记忆，哪些最近变化？
2. 某条记忆在哪里，内容是什么？
3. 哪些记忆会自动注入，哪些永不注入？
4. 哪些记忆重复、孤立、过期或待审核？
5. 某条记忆来自哪个会话，何时、为何被写入？
6. 自动整理将修改什么，是否可撤销？

### 3.2 产品指标

- 记忆查找成功率 ≥ 95%。
- 1000 条文件列表首次可交互时间 ≤ 1 秒（本地环境，不含首次索引）。
- 搜索输入至结果更新 P95 ≤ 300ms。
- 所有写操作均有成功/失败反馈。
- 删除、合并、改写均可在 UI 中恢复或进入回收站。
- 关键功能可纯键盘完成。

## 4. 推荐信息架构

### 4.1 页面总体布局

```text
┌─────────────────────────────────────────────────────────────────────┐
│ Memory  当前项目 / cwd                         [新建] [导入] [整理] │
│ 126 条 · 8 待审核 · 4 孤立 · 自动注入开启                          │
├─────────────────────────────────────────────────────────────────────┤
│ [搜索记忆........................] [类型▼] [来源▼] [注入▼] [排序▼] │
├───────────────┬─────────────────────────────┬───────────────────────┤
│ 导航          │ 记忆列表                    │ 详情 / 编辑 / 来源     │
│               │                             │                       │
│ 全部 126      │ [feedback] 中文提交信息     │ 标题、元数据、正文      │
│ 待审核 8      │ 更新于 2 小时前             │ 出链 / 入链 / 命中记录  │
│ 自动注入 34   │ 摘要……                      │ [编辑] [禁用注入]       │
│ 永不注入 5    │                             │ [查看来源] [删除]       │
│ raw / wiki    │ ...                         │                       │
│ 回收站        │                             │                       │
├───────────────┴─────────────────────────────┴───────────────────────┤
│ [列表] [图谱] [治理] [沉淀任务]                                    │
└─────────────────────────────────────────────────────────────────────┘
```

桌面端使用三栏；窄屏改为“导航抽屉 + 列表 + 详情全屏层”。图谱不再是默认且唯一视图，而是与列表并列的二级视图。

### 4.2 四个一级视图

#### A. 浏览

默认视图，解决查找、阅读、编辑。

- 左侧：目录/类型/状态导航。
- 中间：可搜索、筛选、排序的文件列表。
- 右侧：详情、元数据、来源、链接、注入记录。

#### B. 图谱

用于发现关系，不承担基础浏览职责。

- 搜索并定位节点。
- 按类型、目录、注入状态筛选。
- 节点颜色图例。
- “只看当前节点邻居”模式。
- 孤立节点列表。

#### C. 治理

集中呈现：

- LLM 整理建议。
- 重复、断链、孤立、过长、缺 frontmatter、低置信度。
- 批量修改预览与 diff。
- 最近变更和回收站。

#### D. 沉淀任务

承接现有自动沉淀能力：

- 最近任务：queued/running/completed/failed/skipped。
- 每个任务的来源会话、抽取方式、候选数、写入数、跳过原因。
- 失败重试。
- 候选批准、编辑、拒绝。

该视图需要后端补充 job/candidate API。

## 5. 核心交互设计

### 5.1 搜索

新增搜索框，使用 `bindSearch()` 的 300ms 防抖能力：`static/components.js:153-167`。

搜索结果卡片展示：

- 标题和相对路径。
- 类型、来源、注入策略。
- 命中正文片段，而不是文件名片段。
- 更新时间与相关性。

交互规则：

- 空查询展示最近更新，而不是空白。
- URL 或本地状态保存查询和筛选条件。
- 输入 `/` 聚焦搜索，Esc 清空。
- 搜索错误在结果区显示“重试”，不只写控制台。

### 5.2 文件列表

列表项至少包含：

```text
[feedback] 协作偏好：提交信息使用中文        自动注入
feedback/commit-language.md
用户要求本项目所有 Git 提交信息使用中文……
更新于 2026-07-14 · 来源：session · 2 入链 / 1 出链
```

支持：

- `updated_at`、标题、类型、相关性排序。
- 多选后批量设置注入策略、移动目录、删除。
- 500 条以上启用窗口化渲染或分页。
- 列表选择与右侧详情联动，避免频繁模态弹窗。

### 5.3 详情与编辑

将当前查看模态升级为常驻详情面板；编辑时提供：

- 文件名/相对路径。
- frontmatter 表单模式与 Markdown 原文模式切换。
- `type/source/tags/confidence/inject` 的结构化编辑。
- 未保存提示。
- 保存中、保存成功、冲突、失败状态。
- 可选预览分栏。

保存必须检查 HTTP 状态，不得在失败时关闭编辑器。建议封装统一请求函数：

```js
async function memoryApi(url, options) {
  const resp = await fetch(url, options);
  const data = await resp.json().catch(() => ({}));
  if (!resp.ok) throw new Error(data.error || `HTTP ${resp.status}`);
  return data;
}
```

### 5.4 整理复核

优化为“低风险自动选中，高风险显式确认”：

| 动作 | 默认选择 | 交互 |
| --- | --- | --- |
| link | 选中 | 展示双向链接变化 |
| refine | 选中 | 展示 diff |
| merge | 不选中 | 展示源文件与目标完整 diff |
| rewrite | 不选中 | 高亮删除/新增内容 |
| delete | 不选中 | 显示信息是否已被覆盖 |

新增：

- 全选低风险、取消全选。
- 按动作筛选。
- 单项编辑新文件名和新内容。
- 显示模型、耗时、分析文件数、被截断文件数。
- 应用后显示可恢复项，不立即只关闭窗口。

前端状态不再把完整 action JSON 塞入 `dataset`，改为模块级状态：

```js
const memoryOrganizeState = {
  actions: [],
  selectedIds: new Set(),
  applying: false,
};
```

### 5.5 图谱

图谱增强：

- 空图时显示引导：新建/导入记忆。
- 加载失败显示重试按钮。
- 节点搜索和居中。
- 图例、筛选、邻居深度。
- 右键或详情面板展示节点元数据。
- 节点数超过阈值时默认只加载高连接节点或采用分层布局。

键盘支持：

- Tab 聚焦图谱。
- 方向键切换邻近节点。
- Enter 打开节点。
- `+/-` 缩放，`0` 重置。
- 将 ARIA 文案迁入 i18n，并增加当前焦点节点的 live announcement。

## 6. 状态模型

建议集中管理 Memory 页面状态，避免目前多个函数直接读写 DOM：

```js
const memoryState = {
  cwd: '',
  view: 'browse',
  loading: false,
  error: '',
  files: [],
  selectedPath: '',
  query: '',
  filters: {
    type: 'all',
    source: 'all',
    inject: 'all',
    directory: '',
  },
  sort: 'updated_desc',
  graph: { loading: false, error: '', nodes: [], edges: [] },
  organize: { loading: false, applying: false, actions: [] },
};
```

不引入框架，继续使用 vanilla JavaScript，但采用：

- `loadMemoryWorkspace()`：并行获取 summary/tree/files。
- `renderMemoryPage()`：按 state 渲染。
- `selectMemory(path)`：加载详情。
- `setMemoryFilter()`：只改变 state 并刷新列表。
- `memoryApi()`：统一错误解析。

## 8. 可访问性与国际化

### 8.1 必须修复

1. 模态增加 `role="dialog"`、`aria-modal="true"`、`aria-labelledby`。
2. 打开模态时保存原焦点，关闭后恢复。
3. 模态内实现 focus trap。
4. 文件名输入框、正文 textarea 增加可见或屏幕阅读器 label。
5. 所有加载/保存/应用结果进入 `aria-live` 区域。
6. 图谱键盘可操作。
7. `memory-status-info`、图谱 ARIA、错误信息全部 i18n。
8. 删除等危险按钮不得只依赖颜色表达风险。

### 8.2 文案规范

避免直接显示技术性状态：

- `0 files` → `暂无记忆`
- `save failed` → `保存失败：<原因>`
- `LLM 未返回有效整理方案` → `未生成可应用的整理建议，现有记忆未被修改`

## 9. 分阶段实施

### Phase 0：可靠性修复（1～2 天）

- 统一 `memoryApi()`。
- 查看、加载、保存、索引、整理全部检查 `resp.ok`。
- 状态栏与图谱 ARIA 国际化。
- 保存失败时保留编辑器内容。
- 页面增加 loading/error/empty 状态。

### Phase 1：浏览工作台 MVP（3～5 天）

- 后端列表补充 path/frontmatter。
- 页面加入搜索、文件列表、详情面板。
- 支持类型、来源、注入策略筛选。
- 图谱移动到独立 tab。

### Phase 2：治理与可解释性（3～5 天）

- 整理 diff、风险分级、批量选择。
- 入链/出链、孤立、断链展示。
- 自动注入 trace 与来源历史关联到文件详情。

### Phase 3：沉淀审核（3～5 天）

- 候选列表和 job 列表。
- approve/edit/reject/retry。
- 聊天 toast 可跳转到具体候选或已写入文件。

### Phase 4：性能与可访问性（2～4 天）

- 大列表窗口化/分页。
- 大图降采样或分层加载。
- 完整键盘操作、focus trap、屏幕阅读器验证。

## 10. 验收清单

### 功能

- [ ] Memory 页默认显示可搜索文件列表，不再只有图谱。
- [ ] 可按目录、类型、来源、注入状态筛选。
- [ ] 点击列表项可查看正文、frontmatter、来源与链接。
- [ ] 保存失败不会丢失编辑内容。
- [ ] 整理动作可查看原文/新文 diff。
- [ ] 删除、合并、改写可恢复。
- [ ] 自动沉淀任务和候选可追踪。

### 状态与错误

- [ ] 所有请求都有 loading/success/error 状态。
- [ ] 无数据、无搜索结果、网络错误是三种不同状态。
- [ ] 同一按钮 busy 时不可重复提交。
- [ ] 页面切换或 cwd 变化时旧请求不会覆盖新状态。

### 可访问性

- [ ] 所有核心流程可仅用键盘完成。
- [ ] 模态焦点不会逃逸，关闭后恢复焦点。
- [ ] 图谱节点可通过键盘选择和打开。
- [ ] 中英文 key 集合一致，无硬编码英文状态。

### 性能

- [ ] 1000 条记忆列表滚动流畅。
- [ ] 搜索 P95 ≤ 300ms。
- [ ] 图谱超过 300 节点时有降级策略。
- [ ] 重复刷新不会重复绑定事件或并发覆盖页面状态。
